This page is the reference for the arguments accepted by reads and (where noted) writes.

## Read arguments

Accepted by `findMany`, `findFirst`, `findOne`, and `findUnique`:

<TypeTable
	type={{
		where: {
			description: 'Filter for which rows qualify. Structured input or a Drizzle SQL fragment.',
			type: 'WhereInput | SQL',
		},
		select: {
			description: 'Projection: only the listed scalar/relation fields survive. Narrows the result type.',
			type: 'SelectInput',
		},
		include: {
			description: 'Keep all scalar columns and attach the listed relations.',
			type: 'IncludeInput',
		},
		orderBy: {
			description: 'Sort order. One field map, or an array for multi-column ordering.',
			type: '{ [column]: "asc" | "desc" } | [...]',
		},
		take: {
			description: 'Max rows to return. A negative value reverses ordering and takes from the end.',
			type: 'number',
		},
		skip: {
			description: 'Rows to skip from the start.',
			type: 'number',
		},
		cursor: {
			description: 'Cursor position (scalar column values) for cursor-style reads.',
			type: 'CursorInput',
		},
		lock: {
			description: 'Row lock clause for PostgreSQL/MySQL reads (`FOR UPDATE`, `FOR SHARE`, `SKIP LOCKED`, `NOWAIT`).',
			type: 'LockOption',
		},
		meta: {
			description: 'Custom metadata forwarded to hooks and plugins.',
			type: 'Record<string, unknown>',
		},
	}}
/>

`count` and `exists` accept only `where`, `cursor`, and `meta`.

`findUnique` is the strictest of the read helpers semantically, but it still accepts the same query-argument shape as the other reads. The main difference is intent: use it when the `where` clause targets a unique lookup.

## Row locks

```ts
await client.transaction(async (tx) => {
	return tx.users.findMany({
		where: { id: { gt: 0 } },
		lock: {
			mode: 'update',
			skipLocked: true,
		},
	});
});
```

- Supported on PostgreSQL and MySQL read helpers only.
- `count`, `exists`, and all write operations do not accept `lock`.
- `skipLocked` and `noWait` are mutually exclusive.
- `tables` is PostgreSQL-only.
- General relation loading with `include` / relation `select` is intentionally rejected when `lock` is present.

## Filter operators

Operators available inside `where`, by column type. A bare value means `equals`; `null` matches nullable columns.

| Column type | Operators |
| --- | --- |
| String | `equals`, `in`, `notIn`, `contains`, `startsWith`, `endsWith`, `mode`, `not` |
| Number / bigint / Date | `equals`, `in`, `notIn`, `lt`, `lte`, `gt`, `gte`, `not` |
| Boolean | `equals`, `not` |
| Logical | `AND`, `OR`, `NOT` |
| To-one relation | `is`, `isNot` |
| To-many relation | `some`, `every`, `none` |

`mode` is `'default'` or `'insensitive'` (case-insensitive string matching). See [filters](/docs/querying/filters) and [relations](/docs/querying/relations) for examples.

## Projections

```ts
// select — narrows, scalars + relations
{ select: { id: true, author: { select: { name: true } } } }

// include — keeps all scalars, adds relations
{ include: { author: true } }
```

A relation value can be `true` or a nested read-args object (with its own `where`, `select`, `orderBy`, `take`). See [selecting fields](/docs/querying/selecting-fields).

## Pagination arguments

`paginate` accepts everything above plus:

<TypeTable
	type={{
		limit: {
			description: 'Max rows per page.',

		},
	}}
/>

It returns `{ data, pagination: { type, page, perPage, total, pageCount, hasNext, hasPrevious } }`.

`cursor` accepts everything above plus:

<TypeTable
	type={{

		},

			description: 'Cursor token to read rows after.',

It returns `{ data, pagination: { type, hasNext, hasPrevious, nextCursor, previousCursor } }`. See [pagination](/docs/querying/pagination).

## Write-specific arguments

Read args are only part of the API surface. The write methods add a few important options that deserve their own reference.

### Single-row writes

| Method | Extra args |
| --- | --- |
| `create` | `data`, `skipDuplicates?`, `select?`, `include?`, `meta?` |
| `update` | `where`, `data`, `select?`, `include?`, `meta?` |
| `delete` | `where`, `select?`, `include?`, `meta?` |
| `upsert` | `where`, `create`, `update`, `select?`, `include?`, `meta?` |

### Batch writes

| Method | Extra args |
| --- | --- |
| `createMany` | `data[]`, `skipDuplicates?`, `select?`, `include?`, `meta?` |
| `updateMany` | `where?`, `data`, `meta?` |
| `deleteMany` | `where?`, `meta?` |
| `upsertMany` | `data[]`, `target`, `update`, `select?`, `batchSize?`, `where?`, `meta?` |

### `upsertMany` details

```ts
import { sql } from 'drizzle-orm';

await client.users.upsertMany({
	data: [
		{ email: 'a@example.com', name: 'A', active: true },
		{ email: 'b@example.com', name: 'B', active: false },
	],
	target: ['email'],
	update: (ctx) => ({
		name: ctx.excluded.name,
		active: ctx.excluded.active,
		updatedAt: sql`now()`,
	}),
	batchSize: 1000,
	where: sql`users.disabled = false`,
});
```

`update` accepts four shapes:

- `'all'`
- an explicit column list
- an explicit update object
- a function receiving `{ excluded, sql, table }`

`where` must be a Drizzle `SQL` fragment. It is not the same as the normal structured `where` filter used by reads and single-row writes.
